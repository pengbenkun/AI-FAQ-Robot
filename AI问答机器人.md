# 煤炭设计院数据中台AI问答机器人 - 项目方案

## 1. 项目概述

为煤炭设计院数据中台构建一个 AI 问答机器人，用户通过自然语言提问，系统自动将问题转换为 SQL 查询数据库，并返回友好回答。项目核心功能：

- **自然语言转 SQL**：支持口语化提问，准确生成 SQL 并执行。
- **细粒度权限控制**：每个用户只能访问其有权限的数据表。
- **多轮对话与历史记忆**：支持上下文理解，保存对话历史。

**技术路线**：LangChain + 阿里云线上大模型（通义系列） + 本地 MySQL（业务库+权限历史库） + 本地 Chroma 向量库（表结构索引）。所有数据与向量库均部署在本地，大模型能力通过阿里云 DashScope API 获取，确保数据安全且快速落地。

## 2. 技术栈明细

| 组件              | 技术选型                                  | 说明                                   |
|-------------------|-------------------------------------------|----------------------------------------|
| 后端框架          | FastAPI                                   | 轻量异步框架，提供 REST API            |
| AI 编排框架       | LangChain (0.3+)                          | 使用 LCEL 构建处理流水线                |
| LLM 接入          | `langchain-qwq` + DashScope API            | 调用阿里云通义系列模型                  |
| 通义模型          | `qwen-max-latest` 或 `qwen3-32b-instruct` | 推荐使用最新稳定版                      |
| 嵌入模型          | `text-embedding-v3`                        | 阿里云官方嵌入模型，用于向量化表结构    |
| 向量数据库        | Chroma (本地持久化)                        | 存储表结构向量，无需单独服务             |
| 业务数据库        | MySQL (本地)                               | 数据中台原库，通过 SQLAlchemy 连接       |
| 权限/历史库       | MySQL (与业务库同实例或独立)                | 存储用户权限、对话历史                    |
| SQL 解析          | `sqlparse`                                 | 提取 SQL 中的表名用于权限校验             |
| 开发语言          | Python 3.10+                               |                                          |

## 3. 系统架构

用户请求 → FastAPI → LangChain 编排引擎  
        ├─ 意图识别 (LLM)  
        ├─ 查询改写 (LLM + 历史记忆)  
        ├─ 表结构检索 (Chroma)  
        ├─ NL2SQL (LLM + 动态 Few-shot)  
        ├─ SQL 解析与权限校验 (sqlparse + 本地 MySQL 权限表)  
        ├─ SQL 执行 (SQLAlchemy 连接业务 MySQL)  
        └─ 结果整理与历史记录 (MySQL)

所有组件均部署在同一本地环境，大模型通过 API 调用，数据不离开内网。

## 4. 模块详细设计

### 4.1 意图识别

- **功能**：区分用户问题是“数据查询”还是“闲聊”，避免无效 NL2SQL 调用。
- **输入**：用户问题。
- **输出**：分类结果（`query` 或 `chat`）。
- **实现方式**：通过 LangChain 的 PromptTemplate 调用 LLM，使用简单的提示词让模型判断意图。若为闲聊，则直接返回固定问候语；若为查询，则继续后续流程。
- **提示词设计**：明确要求只输出指定标签，不附加解释，确保解析简单。

### 4.2 查询改写

- **功能**：将口语化、指代不清的问题改写为规范、完整的问题，便于后续检索和 SQL 生成。
- **输入**：
  - 当前用户问题
  - 最近 N 轮对话历史（从历史记录中获取）
- **输出**：改写后的自然语言问题。
- **实现方式**：通过 LangChain 的 PromptTemplate 调用 LLM，结合历史上下文进行改写。提示词中要求保留所有关键实体（如设计院名称、年份、项目类型），只输出改写后的问题。
- **记忆管理**：使用 LangChain 的 `RunnableWithMessageHistory` 或自定义函数从 MySQL 历史表中获取最近几轮对话，拼接成字符串传给 LLM。

### 4.3 表结构向量检索

- **功能**：从业务数据库中召回与用户问题最相关的表结构，为 NL2SQL 提供准确的上下文。
- **数据准备**：
  - 定期扫描业务 MySQL 所有表，获取表名、列名、数据类型、列注释、表注释。
  - 将每张表的元信息拼接成一段描述文本（例如：“表名：project，表注释：项目信息表，列：id(int, 主键)，name(varchar, 项目名称)，start_date(date, 开始日期)”）。
  - 使用阿里云 `text-embedding-v3` 模型为每张表的描述文本生成向量，存入 Chroma 向量数据库，并持久化到本地目录。
- **在线检索**：
  - 将改写后的问题通过同一嵌入模型转为向量。
  - 在 Chroma 中执行相似度搜索，召回 Top-K（如 5-10 张）最相关的表。
  - 返回这些表的描述文本，供 NL2SQL 使用。
- **更新机制**：当业务数据库表结构发生变化时，需重新执行准备脚本，更新 Chroma 索引。

### 4.4 权限管理

- **功能**：确保用户只能访问其有权限的数据表。
- **数据模型**：在 MySQL 中建立权限表 `user_table_permission`，包含字段：`user_id`、`table_name`、`permission_type`（默认为 SELECT）、`created_at`。
- **权限校验流程**：
  1. **预检**：用户请求进入后，从权限库查询该用户所有有权限的表名列表，存入缓存（如内存字典），用于后续快速比对。
  2. **细粒度校验**：NL2SQL 生成 SQL 后，使用 `sqlparse` 解析出 SQL 中涉及的所有表名。逐一比对预检列表，若存在未授权的表，则拒绝执行，并返回“无权限访问某表”的提示。
- **注意**：权限表由管理员维护，可在项目初期手动插入数据，后续可开发管理界面。

### 4.5 NL2SQL 生成

- **功能**：根据检索到的表结构描述、相关示例和用户改写后的问题，生成可执行的 SQL 查询语句。
- **输入**：
  - 检索到的表结构描述文本（多张表拼接）。
  - 动态选择的 Few-shot 示例（与当前表相关的典型问题-SQL 对）。
  - 用户改写后的问题。
- **输出**：SQL 语句（仅 SQL，无额外解释）。
- **实现方式**：
  - 通过 LangChain 的 PromptTemplate 构造提示词，调用 LLM。
  - 使用结构化输出（如 Pydantic 模型）确保 LLM 返回的格式符合预期，避免多余文本。
  - 示例数据可预存在文件或数据库中，根据检索到的表名筛选出相关示例，提升准确性。
- **提示词设计**：强调只能生成 SELECT 查询，禁止修改数据；给出清晰的表结构和示例；要求只输出 SQL。

### 4.6 SQL 执行

- **功能**：执行生成的 SQL，获取查询结果。
- **连接管理**：使用 SQLAlchemy 创建与业务 MySQL 的连接池，每次执行时获取连接。
- **执行控制**：
  - 设置事务为只读（`SET SESSION TRANSACTION READ ONLY`）。
  - 设置语句超时（如 10 秒），防止慢查询影响性能。
  - 限制返回行数（如 100 行），避免结果集过大。
- **结果格式化**：将查询结果（列名 + 行数据）转换为易读的文本格式，如表格或列表。若结果为空，返回“查询结果为空”。

### 4.7 历史记录

- **功能**：存储用户与机器人的对话历史，用于多轮上下文理解和后续追溯。
- **数据模型**：在 MySQL 中建立历史表 `conversation_history`，包含字段：`session_id`、`user_id`、`role`（user/assistant）、`content`、`sql_text`（仅 assistant 时有）、`created_at`。
- **记录时机**：每次用户提问和机器人回答后，分别插入一条记录。
- **历史获取**：在查询改写前，根据 `session_id` 从数据库查询最近 N 条（如 5 条）历史记录，按时间正序拼接后作为上下文输入。
- **多轮记忆**：LangChain 的 `RunnableWithMessageHistory` 可以简化历史管理，但需自定义会话历史存储函数（从 MySQL 读写）。

## 5. API 接口定义

### 5.1 聊天接口

- **URL**: `POST /chat`
- **Request Body**:
  ```json
  {
    "session_id": "string",  // 会话ID，用于多轮对话
    "user_id": "string",     // 用户标识
    "question": "string"     // 用户问题
  }
  ```
- **Response**:
  ```json
  {
    "session_id": "string",
    "type": "query|chat|error",
    "content": "string",      // 回答内容
    "sql": "string|null"      // 如果生成SQL则返回
  }
  ```

### 5.2 历史查询接口

- **URL**: `GET /history?session_id={session_id}&user_id={user_id}`
- **Response**: 返回该会话最近 20 条历史记录，包含角色、内容和时间。

## 6. 开发环境与部署

- **开发环境**：本地机器，Python 3.10+，依赖通过 `requirements.txt` 管理。
- **配置管理**：使用 `.env` 文件存储环境变量，包括阿里云 API Key、数据库连接串、Chroma 路径等。
- **启动方式**：通过 `uvicorn app.main:app --reload` 运行。
- **持久化目录**：`./data/chroma` 存放 Chroma 索引文件；日志可输出到控制台或文件。

## 7. 注意事项与扩展建议

- **表结构更新**：当业务库表结构变化时，需重新运行表结构索引构建脚本。
- **权限管理**：初期可手动插入权限数据；后续可开发管理界面或集成统一权限系统。
- **SQL 解析增强**：`sqlparse` 提取表名可能不完整，可考虑使用 `sqlglot` 提升解析准确性。
- **Few-shot 示例优化**：示例应覆盖常见查询模式，并随业务发展不断补充。
- **错误处理**：增加 SQL 执行失败后的重试机制，将错误信息反馈给模型重新生成。
- **日志与监控**：使用 Python logging 记录关键步骤，便于问题排查。
- **流式响应**：如需实时输出，可改造为 Server-Sent Events (SSE) 或 WebSocket。
